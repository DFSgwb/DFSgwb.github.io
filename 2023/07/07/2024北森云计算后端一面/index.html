数组和链表的区别

数组是一种顺序存储的数据结构，它的每个元素都是相同的数据类型，通过下标来访问数组中的元素，数组的长度是固定的，数组的插入和删除操作比较耗时，时间复杂度为 O(n)。

链表是一种链式存储的数据结构，它的每个元素都是相同的数据类型，通过指针来访问链表中的元素，链表的长度是不固定的，链表的插入和删除操作比较快，时间复杂度为 O(1)。

根据上述的定义不难发现，数组和链表的区别主要有以下几点：
数组和链表是两种常见的数据结构，它们有以下几点区别：

  1. 存储方式：数组是一种连续的存储结构，元素在内存中按顺序排列，而链表是一种离散的存储结构，元素在内存中可以随机分布。
  2. 插入和删除操作的效率：数组在插入和删除元素时，需要移动其他元素来保持顺序，所以效率较低。而链表在插入和删除元素时，只需要调整指针的指向，效率较高。
  3. 访问元素的效率：数组通过下标访问元素的效率很高，时间复杂度为 O(1)。而链表需要从头节点开始遍历链表，直到找到目标元素，所以访问元素的效率较低，时间复杂度为 O(n)。
  4. 内存空间的使用：数组需要一块连续的内存空间来存储元素，所以在申请内存时需要一次性分配足够的空间。而链表可以动态地分配内存空间，每个节点可以在不同的地方分配，所以更加灵活。
  5. 大小的限制：数组的大小是固定的，一旦申请了固定大小的数组，就不能再改变大小。而链表的大小可以动态地增加或减少。

综上所述，数组适用于对元素的访问频繁，而对插入和删除操作较少的场景。链表适用于对插入和删除操作频繁，而对访问元素较少的场景。

HashMap 的底层实现

HashMap 的底层实现是数组+链表或红黑树，当链表长度大于 8 时，链表会转换为红黑树，当红黑树节点个数小于 6 时，红黑树会转换为链表。

在 JDK 1.7 及之前的版本中，HashMap 底层使用的是数组和链表的结合来实现。具体来说，HashMap 的底层是一个 Entry 数组，每个 Entry 对象包含一个键值对。当添加一个键值对时，首先根据键的 hashCode 值计算出在数组中的索引位置，如果该位置为空，则直接将键值对存放在该位置；如果该位置不为空，则通过比较键的 hashCode 值和键值来判断是否已经存在相同的键，如果有相同的键，则更新其值，如果没有相同的键，则将新的键值对添加到链表的末尾。

在 JDK 1.8 及之后的版本中，HashMap 底层的实现进行了优化，当链表的长度超过阈值（默认为 8）时，会将链表转换为红黑树，以提高查找的效率。红黑树是一种自平衡的二叉查找树，可以在 O(log n)的时间复杂度内进行查找、插入和删除操作。

总的来说，HashMap 的底层实现是一个数组，每个数组元素存放一个链表或红黑树。通过计算键的 hashCode 值来确定键值对在数组中的位置，通过比较键的 hashCode 值和键值来判断是否已经存在相同的键。如果存在相同的键，则更新其值；如果不存在相同的键，则将新的键值对添加到链表或红黑树中。

为什么哈希冲突的链表操作从头插 → 尾插

哈希冲突发生时，即两个或多个不同的键值对被映射到了同一个哈希桶中。为了解决这个冲突，一种常见的方法是使用链表来存储冲突的键值对。
链表操作从头插到尾插的原因如下：

  1. 头插法可以保证插入的时间复杂度为 O(1)。在链表的头部插入新节点，只需要改变指针的指向，不需要遍历整个链表。这样可以节省插入操作的时间。
  2. 头插法可以保持相对的顺序性。当链表中存在多个相同哈希值的键值对时，头插法可以将新插入的键值对放在链表的头部，使得相同哈希值的键值对在链表中的顺序与插入的顺序一致。这样可以方便后续的查找和删除操作。
  3. 头插法可以减少内存的分配。在链表的头部插入新节点，不需要额外分配内存，只需要修改指针的指向即可。而尾插法需要分配新的节点内存，并修改指针的指向。所以从内存分配的角度考虑，头插法更加高效。

综上所述，哈希冲突的链表操作从头插到尾插可以提高插入的效率和顺序性，并减少内存的分配。

前台线程和后台线程有什么区别

前台线程：在程序运行过程中用户可以看到并且可以交互的线程，前台线程通常是由主线程创建的，它们的生命周期与主线程相同，当主线程结束时，前台线程也会结束。前台线程可以访问 UI 线程，更新 UI 界面，响应用户的交互操作。前台线程通常是程序的核心线程，它们的执行时间较长，需要占用较多的系统资源。

后台线程：在程序运行过程中用户看不到并且不能交互的线程，后台线程通常是由前台线程创建的，它们的生命周期与前台线程相同，当前台线程结束时，后台线程也会结束。后台线程不能访问 UI 线程，不能更新 UI 界面，不能响应用户的交互操作。后台线程通常是程序的辅助线程，它们的执行时间较短，需要占用较少的系统资源。后台线程通常用于执行一些较为耗时的操作，如网络请求、文件读写等。在 Java 中可以使用 setDaemon()方法将线程设置为后台线程。这样当所有的前台线程结束时，后台线程也会自动结束。

两者的区别：

  1. 生命周期：前台线程在应用程序的主线程中运行，直到应用程序结束或主线程关闭。后台线程则在应用程序的后台运行，即使主线程关闭，后台线程仍然会继续执行。
  1. 优先级：前台线程的优先级较高，操作系统会优先调度前台线程的执行。后台线程的优先级较低，操作系统会将更多的资源分配给前台线程。
  1. 影响：前台线程的运行会阻塞应用程序的关闭，直到前台线程执行完毕。后台线程的运行不会影响应用程序的关闭，即使后台线程没有执行完毕，应用程序也可以正常关闭。
  1. 用途：前台线程通常用于执行耗时的操作，例如网络请求或计算密集型任务。后台线程通常用于执行一些不需要用户交互或持续运行的任务，例如后台数据同步或日志记录。

在 JAVA 中线程池开辟的是后台线程还是前台线程

在 Java 中，线程池开辟的是后台线程。后台线程是指在程序的主线程结束后，后台线程会自动结束，不会阻止程序的退出。相反，前台线程是指在程序的主线程结束后，前台线程仍然会继续执行，直到完成任务或被显式地停止。线程池中的线程都是后台线程，当所有任务完成后，线程池会自动关闭并终止所有后台线程。

java 有哪些实现并发的方法

Java 中实现并发的方法主要有以下几种：

  1. 多线程：使用 Java 的 Thread 类或者实现 Runnable 接口来创建多个线程，并通过线程的 start()方法启动线程。
  2. 线程池：使用 Java 的 Executor 框架来管理和调度线程池。可以使用 ThreadPoolExecutor 类来创建线程池，并通过 submit()方法提交任务。
  3. 同步机制：使用 synchronized 关键字或者 ReentrantLock 类来实现线程的同步，确保多个线程之间的访问顺序和数据的一致性。
  4. 线程间通信：使用 wait()、notify()和 notifyAll()方法来实现线程之间的通信，可以通过等待和唤醒机制来控制线程的执行顺序。
  5. 并发集合：Java 提供了一些并发安全的集合类，如 ConcurrentHashMap、ConcurrentLinkedQueue 等，可以在多线程环境下安全地进行操作。
  6. 锁机制：Java 提供了一些锁机制，如 ReentrantLock、ReadWriteLock 等，可以实现更细粒度的线程同步和控制。
  7. 原子类：Java 提供了一些原子类，如 AtomicInteger、AtomicLong 等，可以实现线程安全的原子操作。
  8. 并发工具类：Java 提供了一些并发工具类，如 CountDownLatch、CyclicBarrier、Semaphore 等，可以实现线程之间的协作和同步。

以上是常见的一些实现并发的方法，根据具体的需求和场景，可以选择合适的方法来实现并发。

mysql 索引有哪些层级

MySQL 索引有以下层级：

  1. 聚集索引（Clustered Index）：每个表只能有一个聚集索引，它决定了表中数据的物理存储顺序。聚集索引的叶子节点包含了整行数据。

  2. 非聚集索引（Non-Clustered Index）：一个表可以有多个非聚集索引，它们的叶子节点不包含整行数据，而是包含了索引键和指向相应数据行的指针。

  3. 唯一索引（Unique Index）：唯一索引要求索引列的值唯一，可以是聚集索引或非聚集索引。

  4. 主键索引（Primary Key Index）：主键索引是一种特殊的唯一索引，用于唯一标识表中的每一行数据。一个表只能有一个主键索引。

  5. 外键索引（Foreign Key Index）：外键索引用于建立表与表之间的关系，它可以加速关联查询。

  6. 全文索引（Full-Text Index）：全文索引用于全文搜索，可以对文本内容进行高效的搜索。

  7. 空间索引（Spatial Index）：空间索引用于存储和查询具有空间维度的数据，如地理位置信息。

  8. 哈希索引（Hash Index）：哈希索引是一种特殊的索引结构，它使用哈希函数将索引键映射为一个哈希值，然后使用哈希值来查找数据。哈希索引只支持等值查询，不支持范围查询。

这些索引层级可以根据具体的需求来选择使用，以提高查询性能和数据的完整性。

Mysql 的最左匹配原则

MySQL 最左匹配原则是指在多列索引中，MySQL 只能使用索引的最左边的列开始匹配，如果查询条件中不包含最左边的列，那么索引将不会被使用。

例如，有一个多列索引 (col1, col2, col3)，如果查询条件中只包含 col2 和 col3，而没有 col1，那么这个索引将不会被使用。因为 MySQL 只能从索引的最左边开始匹配。

这个原则的目的是为了提高查询效率，因为最左匹配可以减少索引的搜索范围，从而加快查询速度。所以在设计索引的时候，需要根据实际的查询条件来决定最左边的列是哪个，以便能够最大限度地利用索引。

需要注意的是，最左匹配原则只适用于联合索引，对于单列索引没有影响。另外，如果查询条件中有范围查询（比如 BETWEEN、>、< 等），那么只有在范围之前的列才能被用于最左匹配。

java 垃圾回收算法

Java 垃圾回收算法是指在 Java 虚拟机中，用于自动回收无用对象的算法。Java 垃圾回收算法的主要目的是释放内存空间，避免内存泄漏和内存溢出等问题。

Java 垃圾回收算法主要有以下几种：

标记-清除算法（Mark and Sweep）：该算法分为标记和清除两个阶段。在标记阶段，垃圾回收器会遍历所有的对象，并标记出所有需要回收的对象。在清除阶段，垃圾回收器会清除所有被标记的对象，并释放它们所占用的内存空间。该算法的缺点是会产生内存碎片，影响程序的性能。

复制算法（Copying）：该算法将内存空间分为两个区域，每次只使用其中一个区域。当一个区域被占满时，垃圾回收器会将所有存活的对象复制到另一个区域中，并清除原来的区域。该算法的优点是不会产生内存碎片，但缺点是需要额外的内存空间。

标记-整理算法（Mark and Compact）：该算法分为标记和整理两个阶段。在标记阶段，垃圾回收器会遍历所有的对象，并标记出所有需要回收的对象。在整理阶段，垃圾回收器会将所有存活的对象移动到内存的一端，并清除另一端的所有对象。该算法的优点是不会产生内存碎片，但缺点是需要移动对象，影响程序的性能。

分代算法（Generational）：该算法将内存空间分为多个代，每个代的对象具有不同的生命周期。新创建的对象会被分配到年轻代，年轻代满了之后会触发一次垃圾回收，将存活的对象复制到存活区。存活区满了之后会触发一次垃圾回收，将存活的对象移动到老年代。老年代满了之后会触发一次 Full GC，清除所有无用对象。该算法的优点是可以根据对象的生命周期进行优化，提高程序的性能。

以上是 Java 垃圾回收算法的主要内容。在实际开发中，需要根据具体的应用场景选择合适的垃圾回收算法，以提高程序的性能和稳定性。
