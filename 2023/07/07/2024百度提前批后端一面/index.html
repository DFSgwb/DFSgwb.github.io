场景题：如何解决菜品超卖问题？

解决菜品超卖问题的方法可以包括以下几个步骤：

  1. 管理库存：定期进行库存盘点，确保库存数量准确无误。同时，建立一个库存管理系统，及时更新库存信息，避免因为库存不准确而导致菜品超卖。

  2. 预测需求：通过历史销售数据和市场趋势进行分析，预测未来的需求量。根据预测结果，合理安排进货和备货，以避免菜品超卖。

  3. 设置限制：在点餐系统中设置每道菜品的库存上限，当库存达到上限时，自动停止接受该菜品的点单。这样可以避免超卖情况的发生。

  4. 提前预订：对于特定的高需求菜品或者特殊场合，可以提前开放预订功能，顾客可以提前预订并支付，以确保能够获得所需菜品，同时也能避免超卖情况的发生。

  5. 及时更新信息：在菜单和点餐系统中，及时更新菜品的库存信息。当某个菜品售罄时，及时将其从菜单中移除或标记为售罄状态，避免顾客点单后才告知缺货的尴尬情况。

  6. 增加供应链合作伙伴：与可靠的供应商建立合作关系，确保及时供货。同时，建立备选供应商的名单，以备不时之需。

  7. 客户沟通：当菜品超卖情况发生时，及时与顾客沟通，提供解释和合理的解决方案，如提供替代菜品或退款等，以维护顾客的满意度。

通过以上措施的综合应用，可以有效解决菜品超卖问题，提升餐厅的服务质量和顾客满意度。

Redis 缓存穿透，击穿，雪崩分别是什么？

Redis 缓存穿透是指恶意请求或者不存在的数据请求不断击穿缓存层，导致请求直接访问数据库，增加数据库的压力，甚至可能导致数据库宕机。

Redis 缓存击穿是指针对某个访问非常频繁的热点数据的请求，无法在缓存中进行处理，紧接着，访问该数据的大量请求，一下子都发送到了后端数据库，导致了数据库压力激增，会影响数据库处理其他请求。原因：经常发生在热点数据过期失效时

Redis 缓存雪崩是指大量数据同时失效

缓解缓存击穿的措施

  1. 设置热点数据永不过期，或者设置过期时间为一个较长的时间，这样可以避免热点数据过期失效时，大量请求直接访问数据库。
  2. 加载缓存时加锁，在加载缓存时，可以使用分布式锁等机制，避免多个请求同时访问数据库
  3. 使用互斥锁：在缓存失效时，可以使用互斥锁等机制，避免多个请求同时访问数据库。
  4. 使用布隆过滤器：在缓存失效时，可以使用布隆过滤器等机制，过滤掉无效的请求，避免大量请求直接访问数据库。
  5. 增加缓存容量：增加缓存容量，可以避免缓存击穿的发生。

缓解缓存雪崩的措施

  1. 设置不同的过期时间，将缓存的过期时间设置为不同的时间，避免同时失效导致大量请求直接访问数据库。
  2. 使用缓存预热，提前加载缓存，避免在缓存失效时，大量请求直接访问数据库。
  3. 使用多级缓存，将缓存分为多级，例如本地缓存，分布缓存等
  4. 使用限流策略：在该并发时，可以使用限流策略，例如漏桶算法，立牌算法等，控制请求的流量
  5. 数据库容灾：在缓存失效时，可以使用数据库容灾机制，例如读写分离，主从复制等，避免数据库宕机。

MySQL 主从复制原理

MySQL 主从复制是指将一个 MySQL 数据库实例（称为主库）的数据复制到另一个 MySQL 数据库实例（称为从库）的过程。主从复制可以用于数据备份、读写分离等场景，提高系统的可用性和性能。

过程：

  1. 主服务器将数据更改记录到二进制日志（binary log）。二进制日志是一个记录了主服务器上所有数据更改操作（如插入、更新、删除）的日志文件
  2. 从服务器连接到主服务器，并请求复制主服务器的数据。从服务器发送一个请求，告知主服务器它希望从哪个二进制日志文件的哪个位置开始复制数据。
  3. 主服务器接收到从服务器的请求后，开始将二进制日志文件中的数据发送给从服务器。从服务器将接收到的数据应用到自己的数据库中
  4. 当主服务器上的数据发生更改时，主服务器将这些更改记录到二进制日志中，并将这些更改发送给所有连接的从服务器。从服务器接收到更改后，将其应用到自己的数据库中。
  5. 从服务器定期向主服务器发送心跳信号，以确保主服务器仍然可用。如果从服务器检测到主服务器不可用，它可以选择连接到另一个可用的主服务器。

但在实际应用中，主从复制过程中可能会出现以下问题：

  1. 主从服务器的数据不一致：主从服务器的数据不一致，可能是由于主从服务器的时钟不同步导致的。解决方法是使用 NTP 网络时间协议，同步主从服务器的时钟。

需要注意的是，MySQL 主从复制的过程中可能会出现延迟、数据不一致等问题。为了避免这些问题，可以采取以下措施：

  1. 设置合适的复制方式：MySQL 主从复制支持异步复制和半同步复制两种方式，可以根据实际情况选择合适的复制方式。
  1. 设置合适的复制延迟：可以通过设置复制延迟来避免从库数据不一致的问题。
  1. 监控主从复制状态：可以通过监控主从复制状态，及时发现和解决问题。
  1. 使用多个从库：可以使用多个从库，避免单点故障和数据不一致的问题。

//快排
public static void quickSort(int[] arr, int low, int high) {
    if (low < high) {
        int index = getIndex(arr, low, high);
        quickSort(arr, low, index - 1);
        quickSort(arr, index + 1, high);
    }
}
/**
 * 获取数组中的分区点
 * @param arr 待排序数组
 * @param low 分区起始位置
 * @param high 分区结束位置
 * @return 分区点位置
 */
public static int getIndex(int[] arr, int low, int high) {
    int pivot = arr[low];
    while (low < high) {
        while (low < high && arr[high] <= pivot) {
            high--;
        }
        arr[low] = arr[high];
        while (low < high && arr[low] >= pivot) {
            low++;
        }
        arr[high] = arr[low];
    }
    arr[low] = pivot;
    return low;
}

// 两数之和 暴力方法
vector<int> twoSum(vector<int>& nums, int target) {
    vector<int> res;
    for (int i = 0; i < nums.size(); i++) {
        for (int j = i + 1; j < nums.size(); j++) {
            if (nums[i] + nums[j] == target) {
                res.push_back(i);
                res.push_back(j);
                return res;
            }
        }
    }
    return res;
}
// 两数之和 哈希表
vector<int> twoSum(vector<int>& nums, int target){
    vector<int> res;
    unordered_map<int,int>mp;
    for(int i=0;i<nums.size();i++){
        if(mp.find(target-nums[i])!=mp.end()){
            res.push_back(mp[target-nums[i]]);
            res.push_back(i);
            return res;
        }
    }
    return res;
}

MySQL 为什么使用 B+树作为索引？为什么不是 B 树，红黑树

MySQL 使用 B+树作为索引的原因有以下几点：

  1. B+树有更好的磁盘读写性能：B+树内部节点不存储数据，只存储索引信息，而叶子节点按照索引顺序存储数据，这种结构可以使得磁盘读写更加连续，减少磁盘寻道时间，提高 IO 性能。

  2. B+树有更好的范围查询性能：B+树的叶子节点形成有序链表，可以很方便地进行范围查询。而 B 树和红黑树则需要在树中进行中序遍历或者回溯来获取范围查询的结果。

  3. B+树有更高的节点利用率：B+树的内部节点不存储数据，只存储索引信息，因此相比于 B 树和红黑树，B+树可以存储更多的索引信息，提高了节点的利用率，减少了树的高度，提高了查询性能。

  4. B+树有更好的顺序访问性能：B+树的叶子节点形成有序链表，可以很方便地进行顺序访问，适合于范围扫描等操作。

综上所述，B+树相比于 B 树和红黑树在磁盘读写性能、范围查询性能、节点利用率和顺序访问性能等方面都有优势，因此 MySQL 选择使用 B+树作为索引结构。
